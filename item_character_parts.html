1. 발견 요약

  • DF 캐릭터 파츠는 영웅(HeroID) 단위 스킨 세트로 관리됨. 부위 단위 독립 교체는 데이터 구조상 지원되지 않음(추정: UI 런타임에서 세트 단위로 교체).
  • 조립 정의의 핵심은 CharacterAvatarData DataTable: 영웅별로 FPP/TPP/UI 뷰에 각 CoreParts[] 배열(AvatarPart 슬롯 → 메시 경로)이 선언됨.
  • 파츠 슬롯은 ECharacterAvatarPartConfig 열거형으로 구분: Upper / Lower / Head / Hair / HairAdaptive (실제 확인 5종).
  • 착용 가능 아이템의 레이어 우선순위는 CharacterAppearanceAdaptationData가 별도 관리 (WearPosition 슬롯 10종 + PriorityAvatar 4종).
  • 영웅 코스메틱(스킨 세트)은 HeroFashionData DataTable: FashionID별로 BelongedHeroID, Class(등급), HeroItemIDs(번들 포함 추가 아이템) 필드.
  • 메시는 일반 품질(Models/Characters/Hero/) 및 HD 품질(Models/Characters_HD/Hero_HD/) 두 세트로 존재.
  • 공유 마스터 스켈레톤: C_Test_001_3P_Skeleton (3P/HD 공통), 영웅 전용 UI 스켈레톤({Hero}_{Skin}_Body_UI_Skeleton)도 스킨별 개별 존재.
  • UI 뷰에서 일부 영웅은 IsUseMasterMesh: true + Master_Man_UI / Master_Female_UI 마스터 메시에 파츠를 LeaderPose 방식으로 붙이는 패턴 확인.
  • ExtraParts에는 ECharacterAvatarExtraTag::SkillItem 타입으로 능력 아이템 메시가 별도 추가됨.

2-2. ECharacterAvatarPartConfig 슬롯 (CoreParts)

실제 JSON에서 확인된 값:

슬롯 코드의미뷰별 사용 여부
Upper상의 / 몸통 메시 (FPP에서는 1P 팔 메시)FPP / TPP / UI 전부
Lower하의 / 하체 메시FPP / TPP / UI 전부
Head얼굴 메시TPP / UI
Hair헤어 메시TPP / UI
HairAdaptive헤어 보조 레이어 (헬멧 착용 시 교체용 추정)TPP / UI (일부 영웅)

FPP(1인칭)에서는 통상 Upper(팔)+Lower(다리) 2슬롯만 선언.
TPP/UI에서는 Upper+Lower+Head+Hair(+HairAdaptive) 최대 5슬롯.

4. 모듈러 조립 방식

4-1. 파츠 합체 방법

실제 JSON 증거에서 확인된 패턴:

TPP/FPP 인게임: IsUseMasterMesh: false → 각 파츠 SkeletalMesh가 공유 스켈레톤 C_Test_001_3P_Skeleton에 바인딩.

  • 모든 3P HD 메시가 동일 스켈레톤 참조 확인 (M_Dragon, F_Hack 등)
  • 파츠 간 포즈 동기화는 공유 스켈레톤 기반 (추정: Leader/Master Pose Component)

UI 뷰 (캐릭터 선택창 등): 일부 영웅에서 IsUseMasterMesh: true + MasterMesh에 Master_Man_UI 또는 Master_Female_UI 지정.

  • Master/ 폴더 내 마스터 메시 종류: Master_1P, Master_Man_3P, Master_Man_UI, Master_Female_UI, Master_Female_LongLeg_UI, Master_Female_LongLeg_ShortShoulder_UI
  • 여성 체형별로 LongLeg, ShortShoulder 변형 존재 (F_Hack 계열 = 기본 Female, F_Luna/F_Zoya 등은 추정 LongLeg)

4-2. 세트 단위 vs 부위 단위

  • CharacterAvatarData의 행 키는 HeroID (영웅+스킨 조합 단위)
  • 하나의 행에 FPP/TPP/UI 전 파츠가 모두 선언 → 영웅 스킨 세트 단위 조립
  • 런타임에서 FashionID 교체 시 해당 영웅의 CharacterAvatarData 행 전체가 교체되는 구조 (추정)
  • 부위 단위 단독 교체 경로는 이 DataTable에서는 미확인

4-3. FaceAnimMasterMesh

  • IsUseFaceAnimMasterMesh: true + FaceAnimMasterMesh 경로가 있는 경우 페이스 애니메이션 전용 마스터 메시 별도 사용
  • Hero_Face/ 폴더: Face_UI_Master_Skeleton, Faces_Master_Skeleton, ABP_Hero_Face_UI, ABP_Hero_M_Face_UI, ABP_Hero_F_Face_UI (성별 구분 ABP 존재)

5. HeroFashionData 스킨·배리언트 구조

"30000050002": {
  "FashionID": "30000050002",
  "WearPosition": 0,               // 착용 슬롯 (0=전신 세트)
  "BelongedHeroID": 88000000025,  // 귀속 영웅 (M_Dragon)
  "IsDefUnlock": false,           // 기본 잠금 (획득 필요)
  "IsDefEquip": false,
  "Class": 5,                     // 등급 (2=기본, 4=??, 5=프리미엄 추정)
  "HeroItemIDs": ["38110050001", "38110050002"],  // 번들 추가 아이템
  "Icon": "/Game/UI/UIAtlas/Texture/Skin/Skin_Wang_IronGrim.*",
  "BundleIcons": [...],
  "DisplayType": 3                // 표시 타입 (2=일반, 3=번들)
}
  • Class: 2 / IsDefUnlock: true = 기본 스킨 (모든 영웅 1개 보유)
  • Class: 4 = 특수 스킨 (SOL 시즌 계열 추정)
  • Class: 5 = 프리미엄 번들 스킨 (HeroItemIDs 포함)
  • FashionID 패턴: 30000020XXX = 기본, 30000040XXX = 특수, 30000050XXX = 프리미엄

9. 불확실 항목

항목상태근거
Leader Pose vs Master Pose 조립 방식추정(LeaderPose)BP(.uasset)이 바이너리라 실제 컴포넌트 구조 미확인. UE5 표준상 CharacterAvatarData IsUseMasterMesh=true → Master Pose Component 유력
WearPosition 슬롯과 AvatarPart 슬롯의 런타임 매핑미확인CharacterAppearanceAdaptationData가 두 슬롯 우선순위 모두 가지나, 런타임에서 어떻게 교차 적용되는지 BP 없이 불명
HairAdaptive의 정확한 트리거 조건추정(헬멧 착용 시 Hair 교체)필드명과 SOL 스킨에만 존재하는 패턴에서 추론
Female 마스터 메시 분기 기준추정(체형 코드)LongLeg/ShortShoulder 이름에서 추론. 어느 영웅이 어느 마스터 사용하는지 AvatarData에서 직접 확인 미완
BodyShapeShiftType 필드 의미미확인모든 확인 샘플이 EHeroShapeShiftType::Zero로 관찰됨. 변형 체형 전환 예약 필드 추정
CharacterProfiles FashionList 항목 ID 범위미확인18000000XXX 형식이 HeroFashionData의 3000XXXXX와 다름. NPC/AI 전용 별도 패션 ID 범위 가능성
HeroFashionData Class 4 등급 의미추정(SOL 시즌권)기본(2)/프리미엄(5) 사이에 위치하고 IsDefUnlock=false인 점에서 추론
M_Elio / M_Flamethrower / M_Rocketeer HD 존재 여부미확인Characters_HD 폴더에서 미발견. FModel 미추출 or HD 없음
