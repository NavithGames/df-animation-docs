5. 적용 방식 분석

5-1. 스킨 적용 방식

BP_Humvee.json 에서 확인:

  • 차량 BP는 DFMVehicleAssembledPartComponent를 사용
  • 각 파츠 컴포넌트는 StaticMesh + MaterialInterface(MI) 직접 참조
  • 스킨 적용은 머티리얼 인스턴스(MI) 교체 방식으로 추정
  • 예: MI_Mobility_Multipurpose_Wheeled_Vehicle_Humvee_body → 스킨 MI로 교체
  • 마스터 머티리얼: M_Weapon_NRMAC_Master_HD (차량/무기 공용 PBR 마스터)
  • VehicleSkin.DefaultSkinAppearanceID → 런타임에 해당 ID의 MI로 스왑하는 구조(추정)
  • Appearance DA/별도 에셋은 추출물에서 미발견 → 런타임 데이터 또는 별도 pak에 존재

5-2. 파츠(Modification) 적용 방식

VehicleModificationRules 필드 분석:

  • SocketType=1 (외형파츠): 메시 교체 방식 추정 — ModelPath 필드 존재하나 추출본에서 null
  • SocketType=2 (기능파츠): 기능성 컴포넌트 교체 (타이어/엔진/무장)
  • AccessoryID != 0: 특정 파츠 선행 장착 필요 (예: 기본 기관총 장착 후 업그레이드 가능)
  • UnlockType=1, UnlockPara=1: 기본 지급 (레벨 무관)
  • UnlockType=2, UnlockPara=N: N레벨 달성 시 해금 (VehicleLevelTable 최대 10레벨)

성능 영향 여부: VehicleModificationRules에 성능 수치 필드 없음. 성능은 별도 VehicleDataConfigTable/VehiclePhysicsTable에서 관리하므로 파츠-성능 직접 연결 테이블은 발견되지 않음 (파츠=외형 전용이거나, 성능 매핑은 PartID로 런타임 처리 추정).

5-3. 전장 실시간 파츠 적용

RealTimeCustomVehiclePartConfigTable + RealTimeCustomVehiclePartConfigExtraTable:

  • 전장 내에서 게임 화폐(500~1500)로 즉시 구매·적용하는 파츠
  • 사전 해금과 별개로 맵별로 제공 파츠가 고정 목록
  • 같은 PartItemID를 로비 커스터마이징과 공유하는 것으로 보임

항목상태근거

파츠 ModelPath (메시 경로)nullVehicleModificationRules의 ModelPath 필드 전체 null — 런타임/별도 pak 추정
