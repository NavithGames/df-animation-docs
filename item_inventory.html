발견 요약

  • 아이템 정의는 단일 대형 ItemDefinition 테이블이 아닌, 아이템 ID 앞자리 prefix 체계로 카테고리/인벤토리 매핑을 구분하는 구조다.
  • 핵심 인벤토리 매핑 테이블 GameItemToInventory(RowStruct: DFMItemToInventoryRuleRow)가 존재하며, ID prefix → 인벤토리 슬롯 번호를 결정한다.
  • 플레이어 장비 프리셋은 EquipmentPresetDataConfig가 Position 코드(슬롯 번호)별 ItemId+수량을 정의하며, 이것이 사실상 인벤토리 레이아웃이다.
  • Extraction 모드(SOL)에 특화된 아이템 드롭 테이블(LootBoxChosenList, LootPointChosenList, DropConfigTable)이 분리되어 있다.
  • 인벤토리 컨테이너 BP 최상위 클래스는 InventoryPickup_DeadBodyWithSpecialEffect(C++ 네이티브, /Script/DFMGameplay)이며, BP는 해당 클래스를 상속한다. 그리드/슬롯 사이즈·무게 등 세부 스펙은 C++ 코드 측에 있어 추출물 JSON에서 직접 확인 불가다.

1-2. 아이템 ID 체계 (GameItemToInventory 기반 역추론)

아이템 ID는 11자리 숫자 또는 앞자리 prefix 형태.
prefix 앞 2자리(MainType)가 카테고리를 결정한다.

[MainType 2자리][SubType 2자리][일련번호 7자리]
예) 10010001471 → MainType=10(주무기), SubType=01, 개별ID=0001471
    11010001003 → MainType=11(헬멧 계열)
    14020000002 → MainType=14(소모품/의료)
    37120100001 → MainType=37(탄약)

GameItemToInventory prefix → Inventory 번호 매핑 (전체 30개 행)

1-4. 아이템 태그/필터 체계 (ItemTagFilterMap)

계층형 태그 구조. MainTag(bIsMainTag=1) + 하위 SubTag(bIsMainTag=0, ParentTagID 참조).

MainTag 0~16 (16개 최상위 카테고리)
  └ SubTag: 101~108 (ParentTagID=1)
  └ SubTag: 201~204 (ParentTagID=2)
  └ SubTag: 301~307, 315 (ParentTagID=3)
  └ SubTag: 401~409 (ParentTagID=4)
  └ SubTag: 501~529 (ParentTagID=5, 가장 많음)
  └ SubTag: 601 (ParentTagID=6)
  └ SubTag: 701~702 (ParentTagID=7)
  └ SubTag: 801~803 (ParentTagID=8)
  └ SubTag: 1601~1610, 1621, 1631 (ParentTagID=16)

총 91개 태그. Desc 필드는 대부분 null (로컬라이제이션 미포함).

2. 인벤토리 구조 및 적재 방식

2-1. 장비 슬롯 코드 (Position) 체계

EquipmentPresetDataConfig CSV에서 확인된 Position 값:

Position 코드추정 슬롯
111주무기 1번 슬롯
101헬멧 슬롯
105방탄조끼 슬롯
107배낭/가방 슬롯

슬롯 코드 패턴: 100번대 = 장비 부위, 107001 = 배낭 내부, 199997 = 예비 탄약.

2-2. 인벤토리 적재 방식

SOL(Extraction) 모드:

  • 아이템을 월드에서 루팅 → InventoryPickup_DeadBodyWithSpecialEffect (C++ 네이티브 클래스) 기반 BP가 처리
  • BP 파생: BP_Inventory_DeadBody_Fake, BP_Inventory_JailBreakDeadBody 등
  • 루팅 인터랙션 태그: Interactor.Pickup.DeadBody, Interactor.Pickup.DeadBody.StartLoot
  • 아이템 배열은 RepItemArray(복제 배열) 구조 → 서버 동기화 방식

드롭 방식 (DropConfigTable):

ItemType enumDropSourceCfg1 예시비고
EDropItemType::DropHelmetLootPoint(30,1),(50,1),(80,1)확률,수량 페어
EDropItemType::DropArmorLootPoint(30,1),(50,1),(80,1)-
EDropItemType::DropWeaponLootPoint(100,1)100% 드롭
EDropItemType::DropConsumeLootPoint(100,1)소모품

Cfg 형식: (확률%, 수량) 페어 복수 지정 가능. 단계별 드롭 확률표.

배틀필드(MP) 모드:

  • EquipmentPresetDataConfig 기반 사전 장비 구성 → 매치 시작 시 로드
  • Position 슬롯 코드 + ItemId + ItemNum 조합으로 정적 장비 세트 구성
  • 인게임 탄약 구매/갱신: ConsumableID(화폐) + RefreshPrice

LootStyle (MatchLootStyle):
AI 루팅 스타일 6종 확인 (중문 원문):

  1. 跑刀仔 (칼질 러너)
  2. 全装大佬 (풀장비 베테랑)
  3. 好战 (호전적)
  4. 保守loot (보수적 루팅)
  5. 合约风格 (계약 스타일)
  6. 随机应变 (랜덤 대응)

2-3. 그리드/슬롯 사이즈 (width×height)

추출물 JSON에서 직접 확인 불가. 그리드 사이즈 필드(width/height/slots)는 C++ 네이티브 클래스 DFMGameplay 내부에 있으며, FModel JSON export 대상이 아니다.
→ 슬롯 기반(Position 코드) 방식임은 확인됨. Tarkov식 그리드 vs 단순 슬롯 여부는 추정 불가.

2-4. 고가치 루트박스 설정 (HighValueLootBoxProp)

  "UniqueID": 1,

4. 불확실/추정 항목

항목상태근거

Fallout/SOL FalloutInventoryItemTable 3개 행확인됨, 불완전추출된 행이 3개뿐. 실제 SOL 아이템 전체 목록은 별도 에셋에 있을 가능성
