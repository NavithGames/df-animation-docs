Rebuild blueprint for WR2 (UE5.7). DF behavior trees are cooked + depend on DF C++ nodes → not directly importable. This is a reconstruction guide, not an asset port. · prepared by Wjna · 2026-06-24

One line: Spot enemy → lock target → find cover & move → blind/aimed fire from cover → reposition if bad spot → evade on danger (grenade / air strike), looped. If no enemy → patrol / idle.

3. WR2 porting rules (important)

  • Engine-default nodes reusable as-is: Selector / Sequence / SimpleParallel / Wait / Cooldown / Loop / RandomSelect / ForceSuccess — all exist in UE5.7 standard BT.
  • DF C++ nodes have no code → must reimplement: DFMAITask_AdvancedMoveTo (→ standard MoveTo + avoidance), GPEnvQueryTest_PathTest (→ standard EQS PathExist test), etc.
  • _C suffix = Blueprint node → recreate same-named BTTask/Decorator/Service BP in WR2 (internal logic is cooked/unreadable, so infer intent from name + connections).
  • Blackboard: DF shares a single BB_AI (few classic BB keys; most state lives in components/code). WR2 — define only the core keys: MainTarget, CoverLocation, CoverContext, CanShoot (bool), SelfState (enum), LastKnownTargetLoc.
  • EQS is the heart of combat: almost every positioning decision (cover / retreat / fine-tune / evade) is EQS. Build the EQS query set first → ~80% of combat quality. Range-band variants (300/600/1200/1500) are engagement-distance tuning; start with 1–2, expand later.

4. Suggested build order for Henrique

  • 1. Blackboard BB_WR2_AI (core keys above)
  • 2. EQS first — 4 queries: FindCoverNearBy, CoverArroundEnemy, MoveBack, TrimPos (+ AvoidGrenade)
  • 3. Core BTTask/Decorator BPs: PVEFightCheck, BaseFightTarget, RunCoverEQS, MoveToCover, Shoot, CheckCanShootTarget
  • 4. Service: UpdateMainTarget
  • 5. Assemble the main BT per the flowchart (BlueprintStack 9347 can create nodes)
  • 6. PIE-verify with one bot → confirm cover / fire / reposition → tune
